A Field Guide · 检索增强生成

别让模型背答案， 让它当场翻书。

你问模型「我们公司试用期多久」，它要么说不知道，要么一本正经地编——因为它的记忆是冻结的、压缩过的、还带知识截止日。RAG 的主张只有一句：把知识从推理里拆出来，用时 检索 → 增强 → 生成。

01Frozen Memory问题

冻结的记忆，关不住的问题

你问一个通用聊天模型「我们公司试用期多久」，它给不出答案——这是私有信息，从没出现在它的训练数据里。模型的全部知识被压进那几千亿个冻结的参数（parametric memory，参数记忆）里：有损、不可更新、还停在某个知识截止日（knowledge cutoff）。

RAG（Retrieval-Augmented Generation，检索增强生成）的主张只有一句：别让模型从记忆里背答案，让它当场翻书。把知识挪到外部，用时检索相关段落、塞进上下文，再让模型基于这些证据作答。

两种记忆 · Parametric vs. Retrieved
参数记忆
知识焊死在权重里。不可更新 · 有截止日 · 会幻觉
检索 →
注入
外部知识库
知识放进可随时换的书架。随改随用 · 可溯源 · 锚在真实文本
RAG 三步：检索 Retrieve → 增强 Augment → 生成 Generate。下面跟着一份三段员工手册，把每一步连数字算给你看。
02Why Not Just Memorize问题

参数记忆的三个洞

为什么需要外挂？因为参数记忆有三个堵不住的洞。

三个洞 · Three Holes
洞 1知识截止 · 训练之后的事一概不知 洞 2私有数据 · 你的内部文档它从没见过 洞 3幻觉 · 被问到边界外不会沉默，会流畅地编
「那微调（fine-tune）一下不就行了？」——不划算：微调贵且慢，知识仍以有损方式压进权重，而且每新增一篇文档就重训一次，根本不现实。RAG 的解法是解耦：知识放进可随时增删的外部库，模型只管推理。改知识 = 改库，不动一根参数。
03Chunking离线建库 · 1/3

离线第一步：把文档切块

问答之前，先把语料离线处理成可检索的形态。第一步是切块（chunking）：把长文档切成小段（典型 200–500 token，相邻块留一点重叠 overlap，防止句子被拦腰截断）。

为什么不整篇存？块太大，命中里混进大量无关内容（召回噪声）；块太小，又会丢失上下文。我们这份手册天然三段，正好一段一块。

04Embedding离线建库 · 2/3

离线第二步：每一块变成一个向量

每块文本喂给嵌入模型（embedding model，如 text-embedding-3、bge），输出一个稠密向量——真实维度通常 768 或 1536。这个向量是文本的语义坐标：意思相近的文本，向量在空间里也相近。

真实的 1536 维不可读，所以我们用一个玩具 3 维空间演示，三个轴假装代表「试用期 / 假期 / 加班」三个语义方向。

嵌入 · 文本 → 语义向量
列方向 = [ 试用期 · 假期 · 加班 ]。看数字：C1 在「试用期」轴上最大，C2 偏「假期」，C3 偏「加班」——几何已经把语义分开了。真实空间是 1536 维、不可读，但相近即相似这条规律完全一样。
05Vector Store离线建库 · 3/3

离线第三步：入库，建近邻索引

把每条 (向量, 原文块) 存进向量数据库（FAISS、Milvus…）。光存还不够——上百万块时逐一比对太慢，于是建一个 ANN 近似最近邻索引（如 HNSW，分层小世界图）：检索从「暴力扫全库」变成沿图「跳几步就到」，毫秒级返回近似 top-k。

06Retrieval在线检索

在线第一步：用余弦找最近的块

用户问 Q「试用期有多久？」。把它用同一个嵌入模型编码（必须同一个，否则两套坐标系对不上），得到查询向量。

然后用余弦相似度（cosine similarity，只看方向不看长度）找最近的几块。手算 cos = Q·C / (|Q||C|)：

检索 · 查询向量找最近邻
Q · C10.99
Q · C20.21
Q · C30.06
取 top-k = 1，C1 胜出（余弦 0.99）。检索的全部，就是这一步——问题向量找最近的知识向量。语义对上了，关键词没对上也能命中。
07Rerank在线检索 · 进阶

命门：检索召回是天花板

这里必须说实话：该出现的块没被捞上来，后面模型再强也救不了，甚至照样幻觉。所以认真的 RAG 会在「召回」和「精排」之间分两段。

两段式检索 · 先广撒网，再精挑
1
粗排 · 召回bi-encoder问题与文档各自编码成向量，比最近邻——快但糙。先捞 top-50 个候选。
2
精排 · 重排cross-encoder把问题和每个候选拼在一起送进模型联合打分——准但慢。只在 50 个里跑，留下 top-5。
稠密向量 · 懂语义 + BM25 · 抓精确关键词 = Hybrid 双路召回
稠密嵌入擅长语义，却会漏精确匹配（产品型号 X-200、人名）。于是常把它和老牌的 BM25（稀疏 / 词频）双路合并，再交给重排——这就是 hybrid search。
08Augment生成

在线第二步：把证据拼进 prompt

把检索到的块插进 prompt 模板——这就是 RAG 里 “Augmented” 的那一步。模型这下不靠记忆了，证据就摆在上下文（in-context）里。

增强 · 检索结果填进模板
根据以下资料回答问题，只依据资料，不要编造。
【资料】新员工试用期为三个月，期间双方均可提前三天通知解除合同。
【问题】试用期有多久？
高亮那一行就是检索回来的 C1 原文，被「贴」进资料槽。模型看到的不再是空泛的提问，而是带着出处的证据。

10Limits边界

别神化它：三条边界

RAG 不是把模型变聪明，而是给它一座随时能换书的图书馆：推理留在参数里，知识搬到了库里。但它也有清楚的边界。

边界 · Where It Breaks
边界 1召回是天花板 · 检索不到 = 答不出 边界 2Lost in the Middle · 上下文太长时中段注意力衰减，所以才要重排把最相关的顶上去 边界 3不是万能 · 知识要内化成能力时微调更合适；多跳推理则走 agentic RAG
