A Field Guide · KV 缓存

KV 缓存， 让生成不必从头再来。

你看着模型一个字一个字往外蹦，却几乎不卡。秘密不是什么高深算法，而是一条最朴素的工程直觉用对了地方——别重算不变的东西。它把每一步的计算量，从 \(O(n^2)\) 压到了 \(O(n)\)。

O(n²)→O(n)省下的计算
0.5 MB每 token 显存
2个推理阶段
01Autoregression循环

生成，是一个循环

大模型生成文字是自回归的：每次只预测下一个 token，把它接到序列末尾，再用更长的序列预测再下一个。于是生成一段 100 词的回答，本质上就是把整座网络前向跑 100 遍，而且每一遍的输入，都比上一遍长一个词。

逐词生成 / one token at a time
输入 猫→预测 很
输入 猫 很→预测 累
输入 猫 很 累→预测 …
每一轮，序列只在末尾多一个词（橙色）。前面的词，一字未改——这个不变性，正是后面一切优化的根。
02The Redundancy浪费

朴素做法，在重复劳动

问题来了。第 \(t\) 步要对 \(t\) 个 token 做一次因果自注意力，标准做法是为当前序列里的每一个 token 重新算 Q、K、V。可其中前 \(t-1\) 个，上一步刚刚算过，而且结果分毫不差。把每一步累加，重复计算量是 \(1+2+\cdots+n = O(n^2)\)。

每个格子 = 一次 K/V 计算
第 1 步
猫
第 2 步
猫很
第 3 步
猫很累
虚线灰格 = 与上一步完全重复的计算。橙格才是这一步真正新增的那一个。重复的部分，越往后越多。
03The Frozen Past观察

关键：过去的 K、V 永不改变

为什么「结果分毫不差」？因为在因果注意力里，每个 token 的 K、V 只由它自己那一刻的隐状态决定，而它看不到后面的词——所以它的隐状态一旦算出就被冻住，往后每一步都一样。「猫」的 \(k\)、\(v\) 在第 1、2、3 步里，是同一个向量。

三步里的 k猫，逐位相同
既然每一步都在重算同一个结果，那把它算一次、存下来，不就够了？这就是 KV 缓存的全部动机。
04Why Not Q取舍

那为什么不缓存 Q？

注意一个不对称：每一步你只用当前新 token 的查询 \(q_t\) 去对历史打分；过去那些 query 用完即弃、再不复用。而 K、V 会被未来每一个 query 反复查询。判据很简单——未来还会被用到的，才值得缓存。

谁该被存下来 / what to keep
向量未来还会被用到？缓存吗
Q · 查询否 —— 当前步用完即弃✗ 丢弃
K · 键是 —— 被每个未来 query 打分✓ 缓存
V · 值是 —— 被每个未来 query 取用✓ 缓存
所以名字叫 KV 缓存，而不是 QKV 缓存——Q 天生留不住，也不必留。
05The KV Cache机制

把 K、V 存进缓存

于是机制呼之欲出：开一块内存，按层、按头存下每个 token 的 \(k\)、\(v\) 向量。每生成一个新 token，只算它自己的 \(k\)、\(v\)，追加到缓存末尾，再用它的 \(q\) 对整个缓存做一次注意力。每一步的新计算量，从「整段历史」缩到「一个词」。

缓存的结构 / 一行 = 一个 token
底部橙色行 = 当前步新算并追加的；上方绿色行 = 直接从缓存复用，不再计算。缓存只增不改，每步长高一行。
06Worked Example演算

连数字一起：看缓存长大

把「猫很累」当成一句正在被生成的话（数字沿用上一篇《注意力机制》）。每喂入一个词，缓存只新增一行，其余都是旧行复用。到第 3 步喂入「累」，\(q_{\text{累}}\) 对三行 key 打分——正是上一篇亲手算过的那一步。

缓存逐步追加 / K-cache & V-cache
第 1 步喂「猫」→ 缓存 1 行；第 2 步喂「很」→ 只算 1 行、追加；第 3 步喂「累」→ 又只算 1 行。始终只有底部一行是新计算。
最后一步：用 q累 查询整个缓存
\[ [\,3,\;0,\;2\,]\ \xrightarrow{\ \div\sqrt{d_k}\ }\ [\,1.5,\;0,\;1.0\,]\ \xrightarrow{\ \text{softmax}\ }\ [\,0.55,\;0.12,\;0.33\,] \]
→ 猫0.55
→ 很0.12
→ 累0.33
分数 \([3,0,2]\) 来自 \(q_{\text{累}}\) 与缓存里三行 key 的点积。缓存让这一步不必重算 \(k_{\text{猫}}\)、\(k_{\text{很}}\)——它们早就在缓存里了。
07Prefill vs Decode两阶段

两个性格迥异的阶段

有了缓存，推理自然分成两段。Prefill（预填充）把整个 prompt 一次性并行喂入，一个大矩阵乘就填满整段缓存。Decode（解码）逐 token 生成，每步只追加一行，矩阵乘退化成矩阵×向量。这也是为什么「首字延迟」和「逐字吐字速度」由完全不同的因素决定。

一次填满 vs 逐行追加
Prefill预填充
  • 整个 prompt 并行过一遍
  • 一个大矩阵乘填满缓存
  • 决定首字延迟（TTFT）
compute-bound · 算力密集
Decode解码
  • 每步只算 1 个 token
  • 矩阵乘退化成矩阵 × 向量
  • 瓶颈是反复搬运缓存
memory-bound · 访存密集
Decode 阶段算得很少、却要把整块 KV 缓存从显存反复读进来——于是速度被显存带宽卡住，而不是算力。
08The Memory Cost代价

代价：缓存吃显存

省下计算，代价是显存。缓存大小正比于「层数 × 序列长 × batch」。算一笔真账——Llama-2-7B（32 层、\(d_{\text{model}}=4096\)、fp16），每个 token 的 KV 约 0.5 MB；4096 上下文就是 2 GB，而这只是一条请求。上下文一长、并发一高，KV 缓存就撑爆显存。

缓存大小公式
\[ \text{KV cache} = 2 \times L \times n_{\text{kv}} \times d_{\text{head}} \times \text{seq} \times \text{batch} \times \text{bytes} \]
0.5 MB每 token（7B,fp16）
2 GB4K 上下文 · 单条
线性随 seq 与 batch 增长
同一条请求，缓存随上下文线性膨胀
512 token0.25G
2K token1G
4K token2G
8K token4G
公式里开头的 2，就是「K 和 V 各存一份」。\(n_{\text{kv}}\times d_{\text{head}}\) 合起来就是模型维度——下一步要动的，正是这块。
09Slimming It Down优化

给缓存瘦身的几招

既然瓶颈是缓存的大小，那就让多个 query 头共享同一份 K、V。MQA 让所有头共享一份，缓存直接缩小 \(h\) 倍；GQA 折中——每组头共享一份（Llama-2-70B：64 query 头配 8 KV 头，8× 更小；Llama-3 全系采用）。

K、V 头怎么被共享 / 上排 query 头 · 颜色 = 共享同一份 KV
同色的 query 头，共用一份 K、V。从 MHA 到 MQA，KV 头从 8 缩到 1，缓存也就缩到 1/8。
不止瘦身：还有管理与压缩
1
GQA / MQA共享 KV 头从结构上减少要存的 K、V 份数，今天的开源大模型几乎都用。
2
PagedAttentionvLLM像操作系统分页一样管理缓存，消除碎片、逼近零浪费，吞吐大涨。
3
滑动窗口Sliding Window只保留最近 N 个 token 的 KV（如 Mistral），让缓存不再无限增长。
4
KV 量化int8 / int4把缓存里的数值压到更低精度，用一点精度换一半甚至更多显存。
它们解决的是同一个问题的不同侧面：让那块只增不减的缓存，别那么贵。
10The Big Idea收束

一句话，归位

回头看，KV 缓存没有任何魔法。它只是把一条最朴素的工程直觉——别重算不变的东西——精确地安放在因果注意力的一个结构性事实上：过去的 K、V 不会改变。用一点显存，换掉一整座 \(O(n^2)\) 的计算冗余。

四个节拍 / the whole idea
观察过去的 K、V 冻住不变 缓存存下 K、V，丢掉旧 Q 两阶段Prefill 填满，Decode 追加 代价显存线性增长，于是瘦身
今天你用的每一个大模型，背后都靠这块缓存，把「逐字生成」变得可负担。

从「每步重算整段历史」到「每步只算一个新词」——这一块缓存，撑起了今天所有大模型的流畅吐字。
