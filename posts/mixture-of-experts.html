原理手记 · 混合专家 Mixture of Experts

一屋子专家， 每次只点亮两个。

把臃肿的前馈层拆成许多个小专家（expert），再加一个分诊台（router）：每个 token 只惊动其中一两个。于是总参数可以涨十倍，单次算力却几乎不动——总参数（容量）和活跃参数（算力）从此解绑。

01The Thesis · Decoupling

把"什么都会"换成"各有所长"

Scaling law 说模型越大越聪明。可问题在于：稠密（dense）模型里，每多一个参数，都要在每个 token、每次前向上付一次算力。参数与算力被死死绑在一起——想要十倍知识，就得付十倍计算。

混合专家做的事，一句话：把总参数和单次算力解耦。它把最吃参数的那一层拆成许多个小专家，再加一个分诊台，让每个 token 只去找其中一两个。于是参数可以铺得很大（这是容量），而一个 token 实际惊动的计算几乎不变（这是算力）。

两个数字，一辈子要分清
总参数Total · 决定知识容量
活跃参数Active · 决定单次算力
例：Mixtral 8x7B 总参数 46.7B，但每个 token 只走约 12.9B——推理成本像个 13B 的小模型，肚子里却装着 47B 的知识。MoE 的全部魅力，就是把这两个数字拉开。
02Where It Sits · The FFN

被替换的，只是那一块前馈层

回忆一个 Transformer block：自注意力（attention）+ 前馈层（FFN / MLP）。FFN 先把维度从 \(d\) 升到 \(4d\)、过一个激活，再压回 \(d\)。它通常吃掉一个 block 里约三分之二的参数（注意力四个投影约 \(4d^2\)，FFN 约 \(8d^2\)）。

MoE 只动 FFN 这一块：把一个 FFN 换成 \(N\) 个并排的 FFN（每个叫一个 expert），外加一个 router。注意力照旧人人都过、参数共享。

03The Router · Gating

路由器：谁来接这个 token

router 是个极小的线性层。对 token 的隐藏向量 \(x\)，乘上一个矩阵 \(W_g\) 得到 \(N\) 个分数（logits），再 softmax 成"路由概率"。这是 MoE 唯一新增的"决策器官"，参数量只占模型的零头。

\[ g(x)=\operatorname{softmax}\!\big(x\,\htmlClass{m-q}{W_g}\big),\qquad \htmlClass{m-q}{W_g}\in\mathbb{R}^{\,d\times N} \]

\(g(x)\) 是一个长度为 \(N\) 的概率向量：它对每个专家给出一个"我有多想把这个 token 交给你"的权重。它逐 token 算一次——同一句话里不同的字，会得到不同的分诊结果。

04Top-k · Sparse Activation

Top-2：只叫醒前两名

取分数最高的 \(k\) 个专家（Mixtral 取 top-2；Switch Transformer 干脆 top-1），其余全部丢弃，再把留下的两个重新归一化。最终输出是这几个专家输出的加权和：

\[ y=\sum_{i\in\mathcal{T}}\frac{\htmlClass{m-q}{g_i(x)}}{\sum_{j\in\mathcal{T}}\htmlClass{m-q}{g_j(x)}}\;\htmlClass{m-k}{E_i(x)},\qquad \mathcal{T}=\mathrm{TopK}\big(g(x),\,k\big) \]

对 "猫" 来说：top-2 是 \(E_3(0.63)\) 与 \(E_7(0.22)\)，归一化成 \(0.74\) 与 \(0.26\)。剩下 6 个专家对这个 token 一次都没跑。换一个 token，被点亮的就是另外两个——试着点下面不同的字。

05Almost Free · Compute vs Capacity

为什么这几乎是免费的午餐

如果你把 8 个专家全跑（那就是稠密），算力是 \(8\times\text{FFN}\)。而 MoE top-2 永远只有 \(2\times\text{FFN}\) 的算力——无论你加到多少个专家。

\[ \text{FLOPs}_{\text{token}}\;\propto\; k\cdot\text{FFN} \]

关键的非对称就在这里：每 token 的算力只跟 \(k\) 有关，与专家总数 \(N\) 无关。于是你可以一路加专家（加容量），单次思考成本却纹丝不动。下面这些真实模型，把"总参数"拉成长长一条，而"活跃参数"只占其中一小截。

06It Collapses · Load Balancing

路由会塌缩：负载均衡

放任不管，router 会自我强化地塌缩：少数热门专家拿走几乎所有 token，越被选中越被训练得好，越好就越被选中；剩下的专家饿死，永远学不到东西。典型的赢家通吃。第一件武器是一条辅助负载均衡损失（auxiliary load-balancing loss），轻轻把分布压平：

\[ \mathcal{L}_{\text{aux}}=\alpha\,N\sum_{i=1}^{N} \htmlClass{m-q}{f_i}\,\htmlClass{m-k}{P_i} \]

其中 \(\htmlClass{m-q}{f_i}\) 是实际被分到专家 \(i\) 的 token 比例（硬计数），\(\htmlClass{m-k}{P_i}=\frac{1}{T}\sum_t g_i(x_t)\) 是 router 给它的平均概率（软）。两者都均匀（\(=1/N\)）时这个乘积最小——它专门惩罚"既拿得多、又给得自信"的专家。\(\alpha\) 一般取 \(0.01\)：推一下就好，别盖过主损失。

第二件武器是专家容量（expert capacity）：给每个专家一个固定缓冲区 \(\text{capacity}=f_{\text{cap}}\cdot\frac{T}{N}\)（容量因子 \(f_{\text{cap}}\) 常取 1.0–1.25）。超出的 token 直接丢弃（drop）、只走残差。丢 token 听着可怕，换来的是整齐、可批处理的 GPU 计算。还可叠一条 router z-loss \(\big(\log\sum_i e^{(xW_g)_i}\big)^2\) 约束 logit 别爆，稳住训练。前沿做法更进一步：DeepSeek-V3 用一个可调偏置项做无辅助损失（auxiliary-loss-free）的均衡，绕开 aux loss 对质量的拖累。
07What Do Experts Specialize In?

专家到底在分什么工

"专家"这个名字容易让人误会：你大概会以为 E3 管医学、E7 管法律。可 Mixtral 团队真的去看了路由，结论出人意料——专家分工更像语法/位置层面，而非话题层面。同一篇文章里数学和生物的 token 路由分布几乎一样；真正稳定的规律是结构性的：缩进、标点、连续重复的 token 常被同一个专家接走，相邻位置也倾向走同一个专家。

08Smarter Experts · Shared & Fine-grained

更聪明的专家：共享 + 细粒度

Mixtral 之后，DeepSeekMoE 给出两个升级。细粒度专家（fine-grained）：把每个专家切成更小的几个、然后多选几个——比如不用"8 选 2"，改成 "64 选 8"。活跃参数预算不变，但组合数爆炸，路由更灵活、单个专家能更专一。

共享专家（shared expert）：留一两个永远激活、人人都过的专家，专门兜住"通用常识"，让被路由的专家腾出手去真正专精。DeepSeek-V2（236B / 21B 活跃）、V3（671B / 37B 活跃，256 个路由专家 + 1 个共享，top-8）都用这套组合。

每个 token = 1 个常驻共享专家 + 路由选中的若干细粒度专家

09Across Devices · Expert Parallelism

铺到多张卡上：专家并行

当全部专家装不进一张卡，就把不同专家放到不同 GPU（expert parallelism）。于是前向多了两次跨卡搬运：路由决定每个 token 该去哪 → all-to-all 把 token 发去持有它那个专家的卡 → 专家计算 → 再 all-to-all 发回原处。

10The Ledger · Trade-offs & Timeline

一份账本，一条来路

MoE 不是免费的，它是一笔清晰的交易：用显存、训练复杂度和通信，换固定算力下更大的容量。什么时候该签这笔单？当你被算力 / 延迟卡住、但显存还算富裕、且想要更大容量时——这正是当下前沿大模型大多选 MoE 的原因。

这笔交易，两边各记一笔

赚到

  • 固定推理算力下灌进数倍参数与知识
  • 同等质量训练更省 FLOPs（GLaM ≈ 1/3 能耗）
  • 专家自然分工、各自专精
  • 加容量 ≠ 加延迟，扩展曲线更友好

付出

  • 显存吃满：全部专家都要常驻，哪怕在睡觉
  • 训练更难调：路由塌缩、负载、不稳定
  • 跨卡 all-to-all 通信开销
  • 批内 token 走不同专家，推理实现更复杂；小 batch 利用率低
