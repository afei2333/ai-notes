A Field Guide · 分词 / Tokenizer

模型不识字， 只认数字。

Transformer 内部全是矩阵乘法——它从不认识「世界」，只认识一串整数。把文字切成 token、再翻回文字的这一层叫 tokenizer；它怎么切，悄悄决定了模型的算力、价格，甚至能不能数清一个词里有几个字母。

01Text → Numbers入口

一切的起点：把字变成数

神经网络只会做一件事——对一堆数字做乘加。所以你打的每个字，进网络前都要先被翻译成一串整数 token ID。以「世界」为例：先按 UTF-8 拆成字节，再被合并、查表，成为模型真正读到的东西。

「世界」是怎么变成数字的
文字世界
↓ 按 UTF-8 拆成字节
字节E4B896E7958C
↓ 合并 + 查表
token ID（示意）3008755642
反过来，模型吐出的也是 ID，再由同一张表翻译回文字显示给你。整篇要回答两件事：「世界」凭什么这样切，以及 strawberry 里到底有几个 r。
02Two Dead Ends动机

两条看似显然、却走不通的路

要把文字变成 token，最直觉的两种切法，各自撞上一堵墙。

字符级 vs 词级 · 两种极端

① 字符级 / char-level

Transformer
词表只要几十个，但一个词就被撑成 11 个 token。注意力是 O(n²)，序列越长越烧算力，还得让模型从零学会拼写。

② 词级 / word-level

runrunningrunner<UNK>
词表几百万也装不下所有词，遇到新词、错字、专有名词只能吐 <UNK>；而且 run / running / runner 被当成三个无关的词，学不到它们同源。
一个太碎、一个太僵。真正的解法在两者中间。
03The Subword Sweet Spot思路

子词：在两堵墙之间找平衡

BPE（Byte Pair Encoding，字节对编码）走中间路线：常见词整块当一个 token，生僻词拆成可复用的碎片。词表固定在几万量级，既不爆炸，也几乎不再有 <UNK>。

tokenization→token+ization
英文里一个 token 平均约 4 个字符、约 0.75 个词——常用词基本一词一 token，长尾词才被拆开。
04Training BPE · Setup训练 ①

BPE 怎么学：先看一份小语料

训练不需要神经网络，只需要数数。拿四个词当玩具语料（右上角是词频），每个词先拆成字符、末尾加一个词尾标记 ·（即 </w>），然后统计所有相邻符号对出现了多少次。

玩具语料 · 拆成字符后统计相邻对
出现最多的相邻对是 (e, s)——在 newest 与 widest 里共 10 次。
把所有词里的相邻对按词频加权计数后，(e, s) 拔得头筹——下一步就合并它。
05The Merge Loop训练 ②

合并循环：高频对，反复粘合

规则就三句话：数出最高频的相邻对 → 把它粘成一个新符号 → 重复。频次相同就按字典序取靠前的一对（保证确定性）。看着 est· 这个真实的英语后缀、和 low 这个词，自己从字符里长出来。

每轮：高亮最高频对 → 合并 → 记一条规则
轮次最高频对频次新符号
1e + s10es
2es + t10est
3est + ·10est·
4l + o7lo
5lo + w7low
停在第 5 步只是为了看清楚；真实训练会重复几万次，直到词表到达目标大小。
06The Merges Are the Model产物

训练的全部产物：一张有序的表

BPE「模型」里没有权重、没有梯度——只有一张按学习顺序排好的合并规则表，外加由此得到的词表。用的时候纯粹是查表，而且顺序不能乱。

学到的 5 条规则（顺序即一切）
1
e + s → es第一个被记下的规则
2
es + t → est复用了上一条的 es
3
est + · → est·粘上词尾，长出后缀
4
l + o → lo换一条主线
5
lo + w → low整个词 low 成形
顺序为什么重要？因为后面的合并依赖前面的结果——es 得先存在，才谈得上 es + t。
07Encoding a New Word应用

编码：把表套到没见过的词上

训练里从没出现过 lowest 这个词。没关系——把它拆成字符，照着合并表从头到尾套一遍，能合就合：

lowest · 依次套用 5 条规则
结果lowest·2 个 token
未知词被拆成两个学过、还恰好是真实语素的碎片：low + est·。这就是子词对新词的泛化——既不 <UNK>，又复用了已知部件。
08Byte-level BPE工程

字节级 BPE：为什么再也没有 <UNK>

真实的 GPT 系 tokenizer 不在「字符」上做 BPE，而在字节上——基础字母表就是 0–255 这 256 个字节。于是任何字符串（中文、emoji、乱码）都先变成字节，天生可编码，永远不会 <UNK>。

「世界」的字节级编码
字符世界
↓ 每个汉字 = 3 个 UTF-8 字节
UTF-8 字节E4B896E7958C
↓ BPE 合并 + 查表
token（示意）3008755642
代价：一个汉字要 3 个字节起步，所以同样一句话，中文通常比英文更费 token——更贵，也更快填满上下文窗口。
09Why It Miscounts后果

strawberry 里有几个 r？

模型常答错，不是笨，是根本看不见字母。这个词被切成两三个子词块，每块在模型眼里只是一个不透明的整数 ID；要数 r，它得凭记忆还原每个 ID 背后的拼写、再做字符算术——而它并没有被专门训练去做这件事。

两种视角 · 同一个词
字母视角 strawberry
token 视角 strID 1156（示意）awID 707（示意）berryID 15717（示意）
人一眼数出 3 个 r；模型看到的却是三个不透明的整数。同源的弱点还有多位数加法、玩字母游戏——根子都在：token 不是字母。
